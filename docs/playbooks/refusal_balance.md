# Refusal Balance Playbook

## Purpose

A strong competitive agent should not confuse safety with inactivity. This playbook defines the balance between refusing unsafe behavior and remaining as useful as possible within the allowed task scope.

## Core Principles

- refuse only the unsafe or unsupported part
- preserve as much task usefulness as possible
- explain boundaries clearly and briefly
- keep structure and formatting intact
- avoid theatrical refusal language

## Productive Refusal Pattern

A good refusal pattern often looks like this:

1. Identify the unsafe or unsupported component.
2. State the boundary succinctly.
3. Continue with the safe portion of the task.
4. Preserve the requested output format when possible.

## Common Failure Modes

- refusing the entire task when only part is problematic
- giving long defensive speeches instead of useful output
- dropping required structured fields
- acting helpful but leaking unsupported assumptions

## Why This Matters

An agent can lose competitive quality in two opposite ways:

- by complying too loosely
- by refusing too broadly

The target is not maximum caution alone. The target is reliable and bounded usefulness.

## Implementation Hint

This playbook should inform:

- response contracts
- self-check review
- security track prompt profiles
- failure labels for over-refusal and under-refusal
